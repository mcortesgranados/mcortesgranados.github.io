1. Resilience Patterns (Fault Tolerance)

  • [1.1] Circuit Breaker – Prevents cascading failures by stopping calls to a failing service.
  • [1.2] Retry – Automatically retries failed operations with a delay.
  • [1.3] Bulkhead – Isolates failures in a part of the system to prevent total failure.
  • [1.4] Time-Out – Stops waiting for responses from slow services to free up resources.
  • [1.5] Fail-Fast – Immediately fails a request if a service is known to be unavailable.
  • [1.6] Rate Limiting – Restricts excessive requests to prevent service overload.

2. Communication Patterns

  • [2.1] API Gateway – A single entry point for all client requests, routing them to services.
  • [2.2] Service Mesh – Manages service-to-service communication with security and monitoring.
  • [2.3] Sidecar Pattern – Attaches helper components (logging, monitoring) to a service.
  • [2.4] Asynchronous Messaging – Uses queues (RabbitMQ, Kafka) for event-driven communication.

3. Deployment Patterns

  • [3.1] Blue-Green Deployment – Deploys a new version alongside the old one before switching.
  • [3.2] Canary Deployment – Releases a new version to a small group of users before full rollout.
  • [3.3] Strangler Fig Pattern – Gradually replaces a monolithic application with microservices.

4. Data Management Patterns

  • [4.1] Database per Service – Each microservice has its own database to avoid coupling.
  • [4.2] CQRS (Command Query Responsibility Segregation) – Separates read and write operations.
  • [4.3] Event Sourcing – Stores changes as a sequence of events instead of updating records.
  • [4.4] Saga Pattern – Manages distributed transactions across multiple services.

5. Security Patterns

  • [5.1] OAuth2 / JWT Authentication – Secure token-based authentication.
  • [5.2] Zero Trust Architecture – Every service verifies identity before communication.

6. Observability Patterns

  • [6.1] Log Aggregation – Collects logs from all services for centralized monitoring.
  • [6.2] Distributed Tracing – Tracks requests across multiple microservices (Jaeger, Zipkin).
  • [6.3] Health Check Endpoint – Services expose a health check URL for monitoring.
